1. SYSTEM OVERVIEW
MeddyNet is a diagnostic lab discovery, booking, and logistics platform. The backend serves four distinct frontend applications — User App, Lab Partner Portal, Technician PWA, and Admin Dashboard — over a single FastAPI REST API, with a dedicated WebSocket server for real-time events.

4. AUTHENTICATION & AUTHORIZATION
MeddyNet uses phone-number OTP login via Supabase Auth. No username/password. After OTP verification, the backend issues its own short-lived JWT (access token) and a longer-lived refresh token stored in an HTTP-only cookie.
OTP Login Flow
Step
Action
Details
1
POST /auth/send-otp
Client sends { phone: '+919876543210' }. Backend calls Supabase auth.sign_in_with_otp(). Supabase triggers SMS via MSG91/Twilio. Rate-limited: 5 OTPs per phone per hour (Redis counter).
2
POST /auth/verify-otp
Client sends { phone, otp }. Backend calls Supabase auth.verify_otp(). On success: fetch/create user in PostgreSQL. Issue JWT + refresh token.
3
POST /auth/refresh
Client sends refresh token. Backend verifies against Redis blacklist. Issues new access token + rotates refresh token.
4
POST /auth/logout
Adds refresh token to Redis blacklist (TTL = token expiry). Clears cookie.
JWT Payload Structure
{ "sub": "user-uuid", "role": "user|lab_admin|technician|superadmin",
  "lab_id": "lab-uuid-or-null", "tech_id": "tech-uuid-or-null",
  "iat": 1714000000, "exp": 1714000900 }
Access token expiry: 15 minutes. Refresh token expiry: 7 days (stored in Redis with user_id mapping for server-side revocation).

RBAC is enforced via a FastAPI dependency injected on each protected route:

@router.get('/admin/users', dependencies=[Depends(require_role('superadmin'))])
async def list_users(db: AsyncSession = Depends(get_db)):
    ...
Middleware Chain (per request)
Cloudflare validates SSL + rate limits at edge
FastAPI CORSMiddleware allows configured origins
slowapi RateLimitMiddleware (100 req/min per IP globally)
AuthMiddleware: extracts Bearer token from Authorization header
JWT decoded via python-jose; user context injected into request.state
Route handler runs with user_id, role, lab_id, tech_id available

7. PAYMENT & WALLET SYSTEM
MeddyNet operates as a marketplace. When a user pays for a booking, the platform temporarily holds the lab's share (escrow-like) in a ledger until the test is completed and the refund window passes. Settlements are then released to labs via Razorpay Payouts.
Payment Flow — Step by Step
Step
Code Path
What Happens
1. Create Order
POST /payments/create-order
Calls razorpay.order.create({ amount, currency: 'INR' }). Stores razorpay_order_id in payments table (status=pending).
2. User Pays
Razorpay Checkout (frontend)
User completes payment in Razorpay's checkout. Returns { razorpay_order_id, razorpay_payment_id, razorpay_signature }.
3. Verify Signature
POST /payments/verify
Backend computes HMAC-SHA256 of order_id + '|' + payment_id using Razorpay secret. If mismatch → 400 Fraud. If match → payment confirmed.
4. Split Commission
payment_service.split()
commission = total * lab.commission_rate (default 0.15). lab_amount = total - commission. Both stored in payments row.
5. Update Wallet
wallet_service.credit()
lab_wallets.pending_balance += lab_amount. lab_wallets.total_earned += lab_amount.
6. Ledger Entry
wallet_service.ledger_credit()
Insert into ledger: { type=credit, amount=lab_amount, reference_type=payment, reference_id=payment.id }.
7. Update Booking
booking_service.confirm()
booking.status = 'confirmed'. WebSocket broadcast: BOOKING_CONFIRMED to lab and user rooms.
Refund Flow
Refunds are only possible if the booking status is not 'completed' or if within the 24-hour post-collection window.
def process_refund(payment_id, amount=None):
    payment = get_payment(payment_id)
    refund = razorpay.refund.create({ payment_id, amount or payment.total_amount })
    payment.status = 'refunded'
    payment.refund_id = refund.id
    wallet.pending_balance -= payment.lab_amount  # deduct from pending
    ledger.append({ type='debit', reference_type='refund', amount=lab_amount })
Payout / Settlement Flow
Payouts are triggered either manually by an admin (POST /admin/payouts/batch) or automatically by a Celery cron job that runs daily at 11:00 PM IST.
@celery.task
def process_daily_payouts():
    eligible_labs = query labs where:
        - pending_balance > 0
        - all bookings older than 24h have status=completed
    for lab in eligible_labs:
        razorpay.payout.create({
            fund_account_id: lab.razorpay_fund_account_id,
            amount: lab.pending_balance,
            mode: 'IMPS', currency: 'INR'
        })
        wallet.pending_balance = 0
        wallet.total_paid_out += amount
        ledger.append({ type='debit', reference_type='payout' })
Lab Bank Account Registration
Before any payout can be made, each lab must have a registered bank account. This creates a Razorpay Contact + Fund Account:
POST /labs/me/bank-account
Body: { beneficiary_name, account_number, ifsc, account_type: 'savings|current' }
Backend:
  contact = razorpay.contact.create({ name, email, type: 'vendor' })
  fund_account = razorpay.fund_account.create({
    contact_id: contact.id,
    account_type: 'bank_account',
    bank_account: { name, ifsc, account_number }
  })
  lab.razorpay_fund_account_id = fund_account.id  # stored encrypted

Redis Pub/Sub Pattern
When running multiple Uvicorn workers, a WebSocket connected to worker A needs to receive messages published by worker B. Redis pub/sub solves this:
# Publisher (any worker, after payment verify):
redis.publish(f'booking:{booking_id}', json.dumps({ 'type': 'BOOKING_CONFIRMED', ... }))
# Subscriber (WebSocket manager, all workers):
async def listen_redis():
    async for message in pubsub.listen():
        room_id = message['channel']
        await broadcast_to_room(room_id, message['data'])

10. FILE STORAGE (CLOUDINARY)

Upload Flow
# report_service.py
import cloudinary.uploader
async def upload_report(file: UploadFile, booking_id: str, lab_id: str):
    result = cloudinary.uploader.upload(
        file.file,
        resource_type='raw',           # for PDFs
        folder=f'reports/{lab_id}',
        public_id=f'{booking_id}',
        format='pdf',
        access_mode='authenticated'    # signed URLs only
    )
    return result['secure_url'], result['public_id'], result['bytes']

15. SECURITY CHECKLIST
Authentication & Session
JWT secret is 64+ random characters, rotated every 90 days
Access token TTL: 15 minutes. Refresh token TTL: 7 days
Refresh tokens stored in Redis with server-side revocation on logout
OTP rate-limited: 5 per phone per hour (Redis counter with TTL)
All /admin/* routes require superadmin role AND IP allowlist check
Payment Security
Razorpay signature verified on every /payments/verify call (HMAC-SHA256)
Razorpay webhook signature verified via X-Razorpay-Signature header
payment_id stored as UNIQUE in DB — duplicate webhook events are idempotent
Bank account numbers encrypted at rest (Fernet symmetric encryption)
Lab fund_account_id never exposed in API responses
Data Protection
All connections use TLS 1.2+ (enforced by Cloudflare)
NeonDB connections require SSL (sslmode=require in connection string)
PII (phone, address, DOB) encrypted at rest in NeonDB (column-level AES)
Cloudinary reports use 'authenticated' access — no public URLs
User phone numbers are not returned in lab/technician API responses
API Protection
Global rate limit: 100 requests/min per IP (slowapi)
Auth endpoints: 5 OTP requests/phone/hour (custom Redis counter)
CORS: only configured origins allowed (no wildcard in production)
File upload: max 10MB, MIME type validated (PDF, JPEG, PNG only)
SQL injection: prevented by SQLAlchemy parameterized queries (never raw SQL)
Cloudflare WAF blocks common attack patterns at edge
16. SCALABILITY ROADMAP
Phase
When
Changes
Phase 1 (MVP)
0-1000 users
Current architecture. Single Railway service. NeonDB free tier. Upstash Redis. All manual payouts.
Phase 2
1k-10k users
Add read replica on NeonDB for analytics queries. Enable Celery auto-payouts. Add Redis cache for lab/test listings (5-min TTL). Split Celery workers by queue (payouts vs notifications).
Phase 3
10k-100k users
Add CDN caching for public lab/test endpoints. Database connection pooling (PgBouncer on NeonDB). Dedicated MongoDB instance for analytics. Rate limiting per user (not just per IP). Multi-region Railway deployment.
Phase 4
100k+ users
Consider Kafka for event streaming instead of Redis pub/sub. Read replicas in each city for low-latency lab search. Sharding strategy for bookings table. Full observability with Datadog or Grafana.
API Versioning Strategy
The current API version is /v1/. When breaking changes are required, /v2/ is deployed alongside /v1/ for a 6-month deprecation window. Version is in the URL path (not headers) for simplicity.
